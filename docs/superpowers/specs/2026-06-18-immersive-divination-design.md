# ZenFortune 沉浸式求签视觉改造设计

日期：2026-06-18
状态：视觉方案已确认，待规格审核

## 目标

本次改造只处理两个明确问题：

1. 首页“寻求内心的明晰”区域不再加载失效的外部图片，改为复用项目内已有签筒图片 `/media/fortune-tube-transparent.png`。
2. 求签进行中的动态画面不再限制在中央卡片内，而是覆盖整个浏览器视口，形成全屏水墨仪式场景。

不修改求签接口、结果生成、语音输入、历史记录或计时逻辑。

## 已确认的视觉方向

采用方案 C“视频与水墨融合”：

- 求签开始后，沉浸层使用固定定位覆盖整个视口。
- 原页面、顶部导航和底部栏位于沉浸层下方并被视觉淡化，不在动画期间继续突出显示。
- 现有视频 `/media/fortune-tube-ritual.mp4` 作为全屏动态核心，使用 `object-cover` 填充视口。
- 视频边缘叠加宣纸色雾化、远山剪影、水墨晕染和暗角，使画面成为完整场景，而不是放大的播放器。
- 不使用卡片边框、圆角、独立面板或中央容器来框定视频。
- 当前提示文案和进度条直接悬浮在画面下部，并通过渐变遮罩保证可读性。
- 桌面和移动端使用相同信息层级；允许背景视频按视口比例裁切，但签筒主体和文字必须保持可见。

## 组件与状态边界

### 首页签筒区域

首页现有介绍区域保持布局和文案不变，仅把远程 `<img>` 地址替换为已有的 `FORTUNE_TUBE_IMAGE` 常量。图片继续使用 `object-contain`，避免签筒被裁切。

### 求签前状态

求签页未进入摇签状态时，保留现有法阵、分类节点、语音入口和返回能力，不改变用户操作路径。

### 求签中状态

`isShaking === true` 时渲染独立的全屏仪式层。该层的职责仅包括：

- 播放现有求签视频；
- 渲染水墨氛围叠层；
- 显示 `retrievingText`；
- 显示 `shakeProgress`；
- 阻止底层页面在仪式期间被误操作。

该层不拥有新的业务状态，不复制计时器，也不改变 `startDivinationRitual` 的请求和完成流程。

### 求签完成状态

现有逻辑将 `isShaking` 设为 `false` 后，全屏仪式层退出，继续打开当前结果弹窗。退出使用短时淡出，避免视频瞬间消失造成跳变。

## 层级与交互规则

全屏仪式层高于应用导航和普通弹窗层，但低于最终结果弹窗。建议层级顺序如下：

1. 底层原页面；
2. 全屏视频；
3. 水墨远山、宣纸雾化和暗角；
4. 状态文案与进度；
5. 结果弹窗。

求签进行中不显示返回按钮，也不允许点击视频关闭仪式。用户已经确认采用覆盖整个浏览器视口的方案 A，因此顶部导航和底部栏不保留可见操作入口。

## 资源与异常处理

- 首页和视频海报统一使用 `/media/fortune-tube-transparent.png`。
- 动态画面使用 `/media/fortune-tube-ritual.mp4`，保持静音、自动播放、循环和 `playsInline`。
- 如果视频尚未加载或播放失败，海报图和水墨叠层仍应构成可用的静态仪式画面；文案与进度继续更新，求签业务流程不能被媒体错误阻断。
- 不新增外部图片 URL，不把媒体加载结果作为求签接口成功的前提。
- 尊重 `prefers-reduced-motion`：减少雾气漂移和缩放，但保留必要的进度反馈。

## 响应式规则

- 仪式层使用 `100dvh`，并保留 `100vh` 兼容回退。
- 视频在窄屏使用 `object-cover`，重点保持中央签筒可见。
- 文案区避开移动设备安全区，底部间距包含 `env(safe-area-inset-bottom)`。
- 进度条宽度在移动端不超过视口的约 60%，桌面端设置合理最大宽度。

## 验收标准

1. 首页目标区域不再请求原 Google 外部图片，签筒图标能够从项目本地稳定显示。
2. 点击“开始求签仪式”后，动态画面覆盖完整视口，不出现卡片边框、圆角或中央视频框。
3. 顶部导航、底部栏和底层页面在求签期间不抢占视觉注意力，也不能被误触。
4. 视频、水墨叠层、提示文案和进度条层级正确；文字在明暗画面上均可读。
5. 求签完成后仍按原流程展示结果，不改变 API 调用、历史保存和图片生成逻辑。
6. 视频加载失败时，海报、水墨背景、提示文案和进度仍正常出现。
7. 桌面窄屏、常规移动端和带安全区的移动端均无明显溢出或遮挡。
8. TypeScript 检查和生产构建通过。

## 验证方式

- 静态检查：确认首页目标图片引用本地常量，确认全屏层只由 `isShaking` 控制。
- 自动检查：运行 `npm run lint` 与 `npm run build`。
- 浏览器检查：首页图片加载、求签前状态、全屏求签状态、进度变化、结果切换和视频失败回退。
- 视口检查：至少覆盖移动窄屏和桌面宽屏，并验证底部安全区。

## 非目标

- 不重新设计结果弹窗。
- 不更换求签视频或生成新媒体资源。
- 不改变求签时长、进度算法和服务端接口。
- 不重构当前大型 `App.tsx`，除非为隔离全屏仪式层所需的最小组件抽取。
